# be-spring-kiosk

## [Mission] Kiosk System: Legacy to Modern Architecture Analysis

본 미션은 제공된 레포지토리의 각 브랜치(또는 버전)를 비교하여, 시스템이 어떻게 엔터프라이즈급 아키텍처로 진화하는지 '역설계(Reverse Engineering)' 관점에서 분석하는 것을 목표로 합니다.

📅 실습 미션: 아키텍처 진화의 추적
여러분은 V1부터 V5까지 순차적으로 코드를 검토하며, 각 단계에서 발생한 설계적 고민과 기술적 도입 이유를 스스로 도출해야 합니다.

1. 설계 구조의 변화 (V1 ➔ V2 ➔ V3)
V1에서 V3로 가는 과정은 '자바 언어의 기본기'를 어떻게 실무적으로 풀어내는가를 보여줍니다.
- Mission: 클래스 다이어그램을 직접 그려보고, 객체 간의 의존 관계가 어떻게 재편되었는지 분석하십시오.

2. 프레임워크의 도입 (V3 ➔ V4)
- V3까지의 '순수 자바' 방식과 V4의 'Spring Boot' 방식은 서비스의 패러다임을 바꿉니다.

- Mission: V3의 main 실행 흐름과 V4의 Application 실행 흐름을 비교 분석하십시오.

  - Key Point: * 객체의 생명주기(Lifecycle) 관리 주체가 누구로 변경되었는가?
  - 서비스의 인터페이스(CLI to Web)가 변경될 때, 핵심 비즈니스 로직을 보호하기 위해 어떤 아키텍처 패턴이 사용되었는가?

3. 데이터 영속성 전략 (V4 ➔ V5)
- 애플리케이션은 데이터를 어떻게 기억하고 관리해야 하는가에 대한 해답을 찾습니다.

- Mission: 데이터 저장 방식의 변화가 서비스 코드(Service Layer)에 미친 영향을 분석하십시오.

  - Key Point: * 추상화 계층(Repository)의 도입이 코드의 결합도에 어떤 영향을 주었는가?

  - 데이터의 상태를 객체로 관리할 때와 일반 자료구조로 관리할 때의 생산성 차이를 논하십시오.

📝 제출 요구사항 (디스코드 스레드에 댓글로 업로드)
실습 미션에서 요구하는 분석 작업을 통해 나온 결과물과 함께 다음 질문에 대한 기술적인 답변이 포함된 분석 결과물을 제출해 주세요.

- Refactoring Point: 각 버전에서 "가장 비효율적"이라고 생각되는 코드와, 그것이 다음 버전에서 어떻게 개선되었는지 Code Diff를 통해 설명할 것.

- Framework Rationale: V4에서 Spring Framework를 도입해야만 했던 타당한 기술적 근거를 3가지 이상 제시할 것. (단순히 "웹을 만들기 위해" 이상의 답변을 해 주세요.)

- (Optional) Architecture Insight: 현재 V5의 구조에서 더 나은 확장성(예: 마이크로서비스 전환, 다양한 결제 수단 추가 등)을 위해 개선할 수 있는 설계적 제언을 포함할 것.
